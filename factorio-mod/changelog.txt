Version: 0.16.15
Date: 2026-07-04
  Bugfixes:
    - fac_gather's "approach" state blacklisted only the ONE unreachable ore tile it tried, not the
      whole patch. An entirely unreachable patch (e.g. coal across water from a distant power-plant
      shore) is typically dozens of adjacent 1-tile entities at nearly identical distance, so "find"
      immediately re-picked the NEXT tile of the SAME dead patch -- exhausting a large patch needed
      O(patch size) deadline cycles and could burn the entire 180s Python-side gather() timeout with
      zero gathered (live-caught: coal gather near a far shore stuck in "approach" for the full 180s).
      Now blacklists every tile of the resource within patch range (radius=15) once one proves
      unreachable, live-verified: same test that previously stalled now reaches 10/10 science packs
      with no gather timeout.

Version: 0.16.5
Date: 2026-07-03
  Bugfixes:
    - fac_fuel_group could feed only the NEAREST burner and starve every other one: with less coal
      than needed to fully top every burner, the greedy nearest-first fill drained the whole supply
      on burner #1 before the "out of coal" check even let burner #2 be tried. Now round-robin: every
      reachable burner gets a turn (tracked in a per-round `served` set) before any one gets a SECOND
      visit, so a scarce supply is actually split across all of them instead of hoarded by whichever
      is closest.
    - fac_fuel_group_status and fac_building_place_status could report a stale/empty result (e.g.
      "fueled:0, machines:0" even though real coal WAS delivered) if the whole job finished inside a
      single poll interval: the queue entry was deleted the instant it finished, so the NEXT poll saw
      plain "active:false" with no numbers. The terminal outcome (success/failure/counts) now survives
      until a status poll actually reads and consumes it.
    - fac_building_place_status (used by every place_smart call project-wide) replied
      {"id":.., "status": {...}} but the Python client read fields off the OUTER object instead of
      the nested "status" -- "active" was always undefined/false, so place_smart declared a build
      "placed" after its very FIRST poll (~1s in), long before the companion had even reached the
      site. The building was frequently never actually created. This affected every async placement
      in the whole project's history (furnaces, drills, power plant, lab, assembler lines, ...).
    - Spectator camera (fac_spectate) re-teleported the viewing player to the companion's exact
      position every tick -- 60 discrete snaps/sec with no interpolation, which read as visible
      jitter/stutter. Now uses LuaPlayer.centered_on (engine-interpolated smooth follow), set once
      and only re-asserted if it changes.

Version: 0.16.2
Date: 2026-07-01
  Bugfixes:
    - Mining no longer cheats. The resource harvest (queues.lua start_mining_next) used to take a
      single lump and then entity.destroy the WHOLE ore tile (up to thousands of units) -- nuking
      deposits and wasting everything but the one lump. It now uses native entity.mine{inventory=...}:
      exactly ONE unit into the companion inventory, the tile's amount decrements by one, and the game
      removes the tile only when it is fully mined out. The character MINES, it never destroys.
    - fac_mine_entity (trees/rocks/buildings) likewise switched to native entity.mine{}: wood from
      trees, stone from rocks, the building's item + contents into the inventory. If the inventory is
      full the entity is LEFT INTACT (no silent item loss, no destroy-without-return).
  Features:
    - fac_spectate <player> <companion> and fac_spectate_stop <player>: put a connected player into the
      native spectator controller and auto-follow a companion's camera every tick (server-side, no
      external teleport loop). Lets a human watch the AI play without a character and without interfering.
